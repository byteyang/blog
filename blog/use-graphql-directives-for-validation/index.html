使用 GraphQL Directives 做自定义格式校验

09.07.2020 — GraphQL — 1 min read

背景

前一阵在写 GraphQL 接口集成测试 时，发现仅仅是传参格式校验，就需要编写许多 Case 。例如，更新机器人回复设置接口：

每个 Int 字段都需要校验「是否为非负整数」。但其实 Case 并不止这些，因为每个 Int 字段的合法条件除了「非负」以外，各不相同：

  • reply_strategy: [1,2,3,4]
  • general_reply_switch: [1,2]
  • general_reply_respond_all: [0,1]

String 类型的reply_priority字段，格式更严格，需要是由这四个单词用逗号拼接成的字符串:

  • TASK
  • KEYWORD
  • QA
  • CHITCHAT

遗憾的是，当前绝大部分的接口都没有做严格传参的格式校验。得益于 GraphQL 的强类型，我们可以避免基础类型的参数错误，例如 Int 传成了 String。但以上面的情况为例，如果基础类型正确，值不合法，GraphQL 是无法检测到的。

难道要在每个接口的开头，手动添加传参校验代码？

踩坑: GraphQL Custom Scalars

GraphQL 提供了一下几种基础类型(标量):

  • Int
  • Float
  • String
  • Boolean
  • ID

既然基础的不能满足需求，是否可以自定义类型？正好 GraphQL 提供自定义标量的方式，我们只要照着文档上的方法就能自定义标量啦:

1const myCustomScalarType = new GraphQLScalarType({
2  name: 'MyCustomScalar',
3  description: 'Description of my custom scalar type',
4  serialize(value) {
5    let result
6    // 这里可以对 result 进行任意操作
7    return result
8  },
9  parseValue(value) {
10    let result
11    // 这里可以对 result 进行任意操作
12    return result
13  },
14})

坑在哪儿呢？

自定义标量虽然可以劫持标量的值，但无法传参。这就意味着，我们需要为每一种规则，创建一个标量:

  • Int_Zero_One
  • Int_Zero_One_Two
  • Int_Zero_One_Two_Three
  • ……

解决: GraphQL Directives

既然标量不能传参，那有没有可以传参的方式？有！那就是 Custom Directives —— 自定义指令。

上代码:

1type Book {
2  title: String @length(max: 50)
3}

可以看到，title 的类型依旧是一个基础标量 String，特殊的是在类型后有个 @length(max: 50) 这就是自定义指令。

自定义指令的实现

1. 创建自定义标量

想要劫持字段值，只能想办法在字段类型标量的序列化/反序列化方法中修改。因为标量默认的构造函数无法传参，因此这里新建一个类，去继承标量类，从而能够改写构造函数。

1class LimitedLengthType extends GraphQLScalarType {
2  // 注意这个构造函数，传了额外的参数: maxLength
3  constructor(type, maxLength) {
4    super({
5      name: `LengthAtMost${maxLength}`,
6
7      serialize(value) {
8        value = type.serialize(value)
9
10        // 使用 maxLength 进行校验
11        assert.isAtMost(value.length, maxLength)
12        return value
13      },
14
15      parseValue(value) {
16        return type.parseValue(value)
17      },
18
19      parseLiteral(ast) {
20        return type.parseLiteral(ast)
21      },
22    })
23  }
24}
2. 通过指令接口访问当前字段，将字段类型改写为新的自定义标量

有了可以传参的新标量，接下来的工作就是获取到自定义参数。通过指令接口，我们可以获取指令中的参数，并且通过「将当前字段类型改写为新标量」的方式，将参数传入字段内部，用于校验。

1class LengthDirective extends SchemaDirectiveVisitor {
2  // 指令接口提供了访问当前字段的能力
3  visitInputFieldDefinition(field) {
4    this.wrapType(field)
5  }
6
7  visitFieldDefinition(field) {
8    this.wrapType(field)
9  }
10
11  wrapType(field) {
12    if (
13      field.type instanceof GraphQLNonNull &&
14      field.type.ofType instanceof GraphQLScalarType
15    ) {
16      // 改写字段类型
17      field.type = new GraphQLNonNull(
18        new LimitedLengthType(field.type.ofType, this.args.max)
19      )
20    } else if (field.type instanceof GraphQLScalarType) {
21      field.type = new LimitedLengthType(field.type, this.args.max)
22    } else {
23      throw new Error(`Not a scalar type: ${field.type}`)
24    }
25  }
26}
3. 引入自定义指令

自定义指令可以在 graphql-tools 中的 makeExecutableSchema 中引入：

1const schema = makeExecutableSchema({
2  typeDefs,
3  schemaDirectives: {
4    length: LengthDirective,
5  },
6})

也可以直接在 apollo-server 中引入：

1const server = new ApolloServer({
2  typeDefs,
3  resolvers,
4  schemaDirectives: {
5    length: LengthDirective,
6  },
7})

如此一来，我们就可以利用同样的自定义指令，通过参数控制，对不同的进行格式验证。例如：

  • reply_strategy: Int @validate(min: 1, max: 4)
  • general_reply_switch: Int @validate(min: 1, max: 2)
  • general_reply_respond_all: Int @validate(min: 0, max: 1)

是不是灵活了很多？
